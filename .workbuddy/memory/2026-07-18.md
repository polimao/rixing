## 防休眠(keep-awake)功能代码审查
审查了 `src-tauri/src/main.rs`(set_keep_awake / apply_keep_awake_from_settings) 与 `src/settings.js`、`src/settings.html` 的防休眠实现,对照 `PRD_防休眠.md`。发现的问题:

1. **[高] "保持屏幕常亮"完全未实现**:后端 `caffeinate -i -m -s` 缺 `-d`(display sleep) 断言;前端无 `keepDisplayOn` 控制、settings.html 也无该开关。看视频/演示时屏幕仍会息屏。
2. **[中] 重启后时长不延续**:`apply_keep_awake_from_settings` 调 `set_keep_awake(true,index)` 用 `-t` 拉满完整时长,忽略已保存 `startedAt` 的已用时间 → 中途重启会"多撑"一段;仅当用户重新打开设置窗口时前端才按原 startedAt 纠正。
3. **[中] 到点无通知**:PRD §3.4 要求"轻量通知/Toast",代码仅翻转 enabled 并释放断言,无任何通知。
4. **[中] 托盘入口缺失**:PRD 规定托盘菜单为唯一必需入口,当前仅设置在窗口有此功能。
5. **[低] 僵尸进程**:`child.kill()` 后未 `wait()`,被 kill 的子进程变僵尸直到 App 退出。
6. **[健壮性质疑] 子进程断言孤儿风险**:用独立 `caffeinate` 子进程持有断言,App 异常崩溃时该子进程可能被孤立而继续持有断言,导致系统永不休眠(PRD §4.2/§8 假设"随进程结束自动失效"未必成立)。

建议:改用直接 IOKit `IOPMAssertionCreateWithName`(持有 assertion id,退出/Drop 时 release),并把 `-d` 与 `keepDisplayOn` 打通;时长倒计时后端统一用"剩余秒数"而非完整时长。

## 防休眠修复(用户明确需求:ToDesk 式——系统不休眠,但允许熄屏/锁屏,后台程序不被中断)
按用户澄清重写实现,`cargo check` 通过。改动:
- **后端 main.rs**:`set_keep_awake(enabled, seconds)` 改为收剩余秒数;`caffeinate` 参数从 `-i -m -s` 改为 **`-i -m`**(去掉 `-s` 以兼容电池,绝不加 `-d` 以允许显示器休眠)。新增 `kill_keep_awake_child()`(kill+wait 回收僵尸)。
- **修复重启重置 bug**:`apply_keep_awake_from_settings` 按 `startedAt` 计算**剩余秒数**传给 `-t`,不再重新撑满整段。
- **到点通知**:新增 `notify_keep_awake_ended` 命令(osascript display notification),前端 `tick` 到点时调用。
- **退出兜底**:改为 `Builder::build().run(|_,event| if RunEvent::Exit { kill_keep_awake_child() })`(本版 Tauri 2.11.2 的 `Builder::run` 只收 context,需用 `build()+run(闭包)` 才能挂 RunEvent)。
- **前端 settings.js**:所有 `set_keep_awake` 调用改为传 `seconds`(新增 `keepSeconds`/`keepRemainingSeconds` 辅助),到点触发通知。
- **PRD_防休眠.md** 重写为 v0.2:ToDesk 式语义,删除"保持屏幕常亮"需求,默认时长 1h,技术实现改用 `caffeinate -i -m`,补充时长准确性(剩余秒数)与验收标准。

## settings.html 拆分重构——可行性评估(用户提案)
用户提案:把 settings.html 的侧边栏/拖拽条抽成独立目录,每个功能建独立目录/页面,侧边栏 tab 点击改为"链接跳转",解决"HTML 作为字符串用 JS 写入不好维护"的问题。
现状事实(已读代码确认):
- settings.html 45KB 单文件:.window-drag-bar(拖拽条)+ .content-row(.tabs 侧边栏 + 多个 .tab-panel 功能面板);切换靠 settings.js 的 `switchTab()` 切 `.active`。
- translate 已半拆到 `src/translate/`(translate.js/translate.css + `#translate-mount`),但仍在 settings.html 同一文档里运行。
- settings.js 47KB 承载**所有**功能逻辑(initTiling/initTodoHotkey/initTranslateHotkey/initKeepAwake/renderAnnList/focus music/about…),真正的维护痛点在这里而非 HTML。
- 共享依赖:theme.css(用 `[data-theme="dark"]`)、i18n.js(`window.I18N.applyI18n(root)`)、ui-common.js。
- Tauri:settings 是单个 `WebviewWindow`(600×550,resizable:false,Overlay 标题栏),启动即建常驻;`show_settings_tab` 靠后端 `emit("settings-open-tab")` 给窗口切 tab。
核心约束:**Tauri 一个 WebviewWindow 只对应一个 HTML 文档,无内置路由**。"链接跳转"有 3 种落地,风险差异大:
1. iframe 外壳(用户字面意思):Tauri API(`window.__TAURI__`)在 iframe 子文档里**不一定可用**→ `invoke` 可能失效,风险最高,不推荐。且 `data-tauri-drag-region` 不能在 iframe 内生效(拖拽条必须留在父文档)。
2. SPA 片段路由(推荐首选):外壳留 sidebar/dragbar,.content 区按 `location.hash` 用 `fetch` 注入各功能真实 .html 片段;单一 JS 上下文,主题/国际化自然继承,`settings-open-tab` 事件照用。最顺滑、回归最小。
3. 真·多页 MPA:每个功能是一个完整独立页(含 sidebar/dragbar 片段),侧边栏 `<a href="ann/index.html">` 真整页跳转;Tauri API 在主文档可用(无 iframe 风险),但每次切换整页重载(轻微闪烁+重 init)。
结论:可行。**避开 iframe**;真工程量在拆 47KB 的 settings.js 成各 feature 模块 + 抽出共享 core(loadSettings/saveSettings/resizeToContent/主题应用/实验室 tab 可见性)。增量建议:先拿已半拆的 translate 试水。待用户拍板选 SPA 片段路由 还是 MPA。
